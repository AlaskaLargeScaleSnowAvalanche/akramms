# RAMMS Internals

This is a description of files RAMMS produces, and what each of them are.

* **Top-Level RAMMS:** An IDL program that generates many files.  Some of them are produced out-of-process with the IDL Bridge.  It's not clear what external executables are used here.

* **Core RAMMS:** A compiled C++ program, it is called (at least) twice with different command line arguments for:
  1. Generate the `.xy-coord` file.
  1. Do the actual RAMMS simulation.

  The Core RAMMS executable found in `bin/ramms_aval_LHM.exe` of the RAMMS binary distribution.  It uses a signle core for a single simulation, and apparently needs 32-bit API.  It may be run under Linux using Wine, which may be installed as follows (assuming 64-bit Linux):
  ```
  sudo apt install wine
  sudo dpkg --add-architecture i386 && sudo apt update
  sudo apt install wine wine32 libwine
  ```

# Directory Structure

* `RAMMS/juneau130yFor`: Top level directory (created by `ramms.py`).  `juneau130yFor` is the *scenario name*.

  * `DEM/`, `FOREST/`: Created by `ramms.py`.  Copies of the DEM and Forest GeoTIFF files for RAMMS.
  * `RELEASE/`: Shapefiles created by eCognition post-processor (with snow depth set).  Can be multiple shapefiles; but easiest if there is only ONE shapefile.
  * `DOMAIN/`: Shapefiles created by Domain Finder, corresponds to shapefiles in `RELEASE/`.
  * `run_ramms.bat` / `run_ramms.sh`: Shell script to launch top-level RAMMS (used via ssh from Linux machine).
  * `RESULTS/`: RAMMS does all its stuff in this directory.

     * `lshm_rock.log`: The main log file from the top-level RAMMS run.
     * `juneau1_For/`: one directory here per shapefile?.  Contains files created by top-level RAMMS run, good for ALL domains in the run:

       * `juneau1_For_L300_mu.tif` / `juneau1_For_L300_xi.tif`: The MuXi files.
         * `logfiles/muxi_altlimits.log` / `logfiles/muxi_class.tiff` are related to MuXi generation.
       * `curvidl.tif`, `slope.tif`: Also generated by top-level RAMMS

       * `5m_30L/`: The actual avalanche runs, with 11 files per domain:

         * Files created by top-level RAMMS IDL (in order of creation):
           ```
           4.6M Sep 28 14:29 juneau1_For_5m_30L_10100.xyz
           1.5K Sep 28 14:30 juneau1_For_5m_30L_10100.av2
            174 Sep 28 14:30 juneau1_For_5m_30L_10100.bat

           * `.xyz`: "GridBin" format Altitude file.  Some binary raster format?
           * `.av2`: Main input file for core C++ RAMMS code.  Human-readable text.  Eg:

             ```
           * `.bat`, `.xy-coord`: The core RAMMS C++ executable is used to produce the `.xy-coord` file, and is run as an IDL out-of-process bridge from the top-level RAMMS IDL.  The `.bat` file contains the command line used to do this.

           * `.dom`: A simple ASCII polygon representation of the domain (translated from the Domain shapefile provided to top-level RAMMS).  Format seems to be all on one line, with the last point repeating the first:
             ```
             <n> <p1.x> <p1.y>, ...
             ```
           * `.rel`: The release polygon, same format as `.dom` file.
           * `.var`: A very large binary file.
           * `.out.bat`: Script calls core C++ RAMMS to do the actual work.  When it is run, the following are produced:
             ```
             4.4K Sep 29 13:38 juneau1_For_5m_30L_10100.out.log        
             262K Sep 29 13:38 juneau1_For_5m_30L_10100.out.gz
               35 Sep 29 13:38 juneau1_For_5m_30L_10100.out.end   
             ```
             * `.out.log`: Text log of what happened.  Probably check here to determine if a domain was too small.
             * `.out.gz`: The output, compressed.  (Because the `.out.bat` script applies `gzip` after the fact).  The uncompressed `.out` file contains MAIN zeros, probably some raster format.
             * `.out.end`: A simple file written to signal in an atomic way that this domain is complete.
